# Паттерны проектирования, архитектурные шаблоны и жизненные циклы запросов в eSPX

В архитектуре eSPX применены классические и специализированные паттерны распределенных систем. Они направлены на обеспечение масштабируемости, отказоустойчивости и поддержание согласованности данных между гетерогенными хранилищами (Postgres, ClickHouse, Redis). Под каждым паттерном приведено детальное описание жизненного цикла выполнения запросов и ASCII-схема движения данных.

---

## 1. Шаблон проектирования Transactional Outbox

### Описание
Когда администратор меняет параметры рекламной кампании (например, дневной бюджет), система должна сохранить это изменение в реляционной базе данных (Postgres) и одновременно обновить горячий кэш в оперативной памяти Redis. Запись в два независимых хранилища напрямую приводит к рассинхронизации в случае падения сети или одного из серверов.

### Жизненный цикл запроса мутации и синхронизации
1. **Запись мутации (ACID)**:
   - Администратор сохраняет новые настройки. В рамках одной транзакции Postgres обновляется таблица `campaigns` и вставляется строка события в `outbox_events` со статусом `PENDING`.
   - Транзакция коммитится. Изменения гарантированно сохранены на диске СУБД.
2. **Работа Outbox Worker**:
   - Поток `OutboxWorker` в цикле опрашивает таблицу событий. Конструкция `FOR UPDATE SKIP LOCKED` гарантирует, что параллельно работающие инстансы не вызовут взаимных блокировок.
   - Выбранная пачка переводится в статус `PROCESSING`.
   - Воркер выполняет параллельные пайплайн-запросы к инстансам Redis, обновляя горячие лимиты.
   - При успешном получении подтверждений от всех шардов транзакция завершается переводом статуса в `PROCESSED`.

```text
[ Admin UI ] 
     │  (POST Request)
     ▼
[ Management API ] 
     │  (Start Transaction)
     ├───► Update campaigns/budgets in PG
     ├───► Insert to outbox_events (status='PENDING')
     │  (Commit Transaction)
     ▼
[ Outbox Worker ]  ◄── (Poll every 20ms: SELECT FOR UPDATE SKIP LOCKED)
     │  
     ├───► Set status = 'PROCESSING' in PG
     ├───► Pipeline concurrent HSET/SET write to all Redis Shards
     ├───► Set status = 'PROCESSED' in PG
     ▼
[ Redis Shards (0..3) ] (Edge State Updated)
```

## 2. Шаблон Pinned Worker Pool (Привязка воркеров)

### Описание
При миллионном трафике (RPS) случайное распределение сетевых запросов по потокам операционной системы приводит к вымыванию процессорного кэша L1/L2. Каждому ядру CPU постоянно приходится загружать контексты разных кампаний из DRAM.

### Жизненный цикл обработки трафика на горячем пути
1. **Прием пакета**: Сетевой поток `gnet` считывает данные из TCP-сокета через `epoll`.
2. **DFA-парсинг**: Конечный автомат (DFA HTTP/1.1) без выделения памяти в куче извлекает JSON-структуру запроса.
3. **Маршрутизация на поток**: Идентификатор кампании хэшируется, и на его основе выбирается конкретная очередь (MPSC ring buffer).
4. **Выполнение фильтров**: Goroutine воркера просыпается, забирает запрос из своей персональной очереди и выполняет цепочку проверок `FilterEngine.Check` на локальном ядре CPU.

```text
[ Inbound Network Socket ]
     │ (gnet reads TCP buffer via epoll)
     ▼
[ gnet event loop (DFA parsing) ]
     │ (Hash campaign_id -> select Worker ID)
     ▼
[ Queue Channel (MPSC Ring Buffer) ]  ◄── (Выравнивание кэш-линий: 64-byte padding)
     │ (Read task)
     ▼
[ Pinned Goroutine (Worker Thread) ]  ◄── (Локальность кэша L1/L2 ядра процессора)
     │ (Runs FilterEngine.Check)
     ▼
[ Redis / Memory operations ]
```

## 3. Шаблон Circuit Breaker (Предохранитель)

### Описание
Падение одного из внешних сервисов (например, провайдера рассылки SMS/Email в `Notifier` или одного из шардов Redis) может привести к накоплению запросов, блокировке пула соединений и полному отказу всего приложения.

### Жизненный цикл состояний предохранителя
1. **Closed (Закрыт)**: Запросы отправляются в штатном режиме. Сбои инкрементируют счетчик ошибок.
2. **Open (Открыт)**: При превышении порога ошибок (`REDIS_BREAKER_FAIL_THRESHOLD = 150`) предохранитель открывается. Все последующие запросы мгновенно завершаются ошибкой на клиенте без отправки запроса в сеть, защищая приложение от зависания.
3. **Half-Open (Полуоткрыт)**: По истечении 5 секунд предохранитель пропускает единичный проверочный запрос. Если он завершается успешно, предохранитель возвращается в состояние Closed. Иначе — снова переходит в Open.

```text
                ┌────────────────────────┐
                │         CLOSED         │◄────────┐ (Reset/Recover)
                └────────────────────────┘         │
                     │                             │
                     │ (Error rate > threshold)     │
                     ▼                             │
                ┌────────────────────────┐         │
                │          OPEN          │         │
                └────────────────────────┘         │
                     │                             │
                     │ (Timeout expired)           │
                     ▼                             │
                ┌────────────────────────┐         │
                │       HALF-OPEN        ├─────────┘
                └────────────────────────┘
                     │
                     │ (First test query fails)
                     ▼
                 (Back to OPEN)
```

## 4. Паттерн SettingsWatcher (Динамическое обновление конфигурации)

### Описание
Сервисы горячего пути (трекеры) не могут обращаться к СУБД для чтения настроек при каждом запросе. В то же время оператор должен иметь возможность мгновенно включить аварийный режим или изменить глобальные лимиты.

### Жизненный цикл обновления конфигурации
1. **Инициирование обновления**: Изменение настроек в панели управления записывает новые значения в хэш-таблицу `config:values` в Redis и инкрементирует счетчик `config:version`.
2. **Опрос версии**: Фоновый процесс `SettingsWatcher` раз в 5 секунд считывает только значение `config:version` с Redis.
3. **Атомарная подмена**: Если версия изменилась, воркер вычитывает весь хэш `config:values` и выполняет подмену указателя на локальную структуру настроек в памяти с помощью `atomic.Value`.
4. **Запросы импорта**: Запросы на горячем пути обращаются к конфигурации через атомарный указатель в памяти без блокировок и сетевых запросов.

```text
[ Redis Shard 0 ]
  ├── HASH `config:values`  (Глобальные лимиты и флаг рубильника)
  └── STRING `config:version` (Монотонный счетчик эпохи конфигурации)
          ▲
          │ (Опрос версии раз в 5 секунд)
[ SettingsWatcher (Background loop) ]
          │ 
          ├───► Версия изменилась? ──► Прочитать config:values из Redis
          └───► Атомарная подмена структуры настроек через atomic.Value
          ▼
[ Hot-path Filters (e.g. EmergencyBreakerFilter) ]
          └───► Чтение настроек из памяти хоста (0-alloc, lock-free)
```

## 5. Бюджетный синхронизатор (SyncWorker & Reconciliation)

### Описание
Списание средств с бюджетов кампаний на горячем пути происходит в оперативной памяти Redis для обеспечения субмиллисекундного времени ответа, в то время как реляционный леджер в Postgres обновляется асинхронно фоновыми воркерами.

### Жизненный цикл движения денег
1. **Фиксация в Redis**: При успешной транзакции Lua-скрипт инкрементирует накопленную дельту списания в ключе `budget:sync:campaign:ID` и добавляет ID кампании в Set `budget:dirty_campaigns`.
2. **Асинхронный забор**: Фоновый воркер `SyncWorker` раз в секунду вытягивает ID измененных кампаний с помощью `SPOP` из множества `budget:dirty_campaigns`.
3. **Обновление леджера**: Воркер считывает дельту из Redis и записывает проводку списания в `balance_ledger` Postgres.
4. **Сброс дельт**: После успешного коммита транзакции в Postgres запускается Lua-скрипт, который вычитает зафиксированную дельту из накопленного в Redis ключа `budget:sync:campaign:ID` (исключая потерю параллельно набежавших списаний).

```text
[ Redis Shard (0..3) ]
  ├── `budget:dirty_campaigns` (Множество ID кампаний с дельтами списаний)
  └── `budget:sync:campaign:ID` (Накопленная дельта списания денег в RAM)
          ▲                           ▲
          │ (SPOP campaign ID)        │ (Вычитание дельты после коммита)
[ SyncWorker (Background thread) ] ───┘
          │
          ├───► 1. Чтение накопленной дельты из Redis
          ├───► 2. Запись списания в таблицу balance_ledger в Postgres
          ├───► 3. Запуск Lua-скрипта фиксации/сброса дельты в Redis
          ▼
[ Postgres DB ] ──► (Главная бухгалтерская книга системы)
```
