# Технический отчет по внедрению Distributed Quotas (Фазы 1.4 – 1.7)

## 1. Обзор архитектуры и результатов

В рамках текущего этапа была полностью реализована и протестирована система распределенных финансовых квот (**Distributed Quotas**) в соответствии с планом `REDIS.md` (Фазы 1.4, 1.5, 1.6, 1.7). 

Эта система реализует паттерн «Центральный банк (Postgres) + локальные кассы (Redis-шарды)», перенося финансовые лимиты на шарды кампаний и обеспечивая высокую пропускную способность трекера (RPS) без межшардовой координации и оверхеда на горячем пути.

### Ключевые результаты:
* **QuotaManager (Фаза 1.4):** Разработан фоновый воркер в Management-сервисе, который асинхронно обрабатывает запросы на рефилл квот из Redis-очереди `budget:refill_needed`, резервирует чанки в Postgres и пополняет балансы в Redis с автоматическим откатом транзакций при сетевых сбоях.
* **SyncWorker и Reconciliation (Фаза 1.5):** Реализовано пропорциональное уменьшение зарезервированных сумм (`reserved_amount`) в Postgres при списании реального баланса из Redis. Добавлена автоматическая очистка зависших резервов при падении Redis-шардов (>60с) и мониторинг расхождения (drift) квот.
* **Локальный микро-кэш на трекере (Фаза 1.6):** Создан сверхбыстрый, lock-free, zero-allocation локальный кэш заблокированных кампаний (`LocalQuotaCache`), снижающий нагрузку на Redis при наплыве трафика на выжженные кампании (hot campaigns).
* **Интеграционное тестирование (Фаза 1.7):** Добавлены сквозные тесты, подтверждающие корректность локальной блокировки, рефилла и соблюдения финансового инварианта.

## 2. Детальное описание реализации по фазам

### Фаза 1.4: QuotaManager (Management, cold path)
Фоновый воркер `QuotaManager` (`internal/management/quota_manager.go`) решает две задачи:

1. **Асинхронный рефилл (1.4.1, 1.4.2):**
   * Раз в 100 мс опрашивает каждый Redis-шард с помощью атомарной команды `SPOP` (`SPopN` до 100 элементов за раз), извлекая ID кампаний, у которых квота упала ниже порога (20%).
   * Для каждой кампании генерирует уникальный UUID идемпотентности и вызывает `QuotaRepo.ReserveChunk` в Postgres.
   * При успехе Postgres-транзакции увеличивает `budget:quota:{cid}` в Redis на `QUOTA_CHUNK_SIZE` и удаляет флаг-замок `budget:refill_lock:{cid}`.
   * **Откатоустойчивость:** Если инкремент в Redis завершился ошибкой (таймаут, дисконнект), `QuotaManager` автоматически откатывает резервирование в Postgres через `QuotaRepo.ReleaseChunk` (уменьшая `reserved_amount` обратно), предотвращая утечку бюджета.

2. **Начальная инициализация квот (1.4.3):**
   * Раз в 5 секунд сканирует активные кампании в Postgres. Если у активной кампании в Redis отсутствует ключ `budget:quota:{cid}`, воркер захватывает распределенный замок `budget:refill_lock:{cid}` через `SETNX` на 10 секунд.
   * Внутри замка повторно проверяет существование ключа, резервирует начальный чанк в Postgres и инициализирует квоту в Redis. Это гарантирует, что кампания сразу готова к приему трафика.

### Фаза 1.5: SyncWorker и Reconciliation
1. **Балансировка инварианта при синхронизации (1.5.1):**
   При периодическом сбросе накопленного в Redis спенда в Postgres (`SyncWorker.syncEntity` -> `CampaignRepo.UpdateSpend`), транзакция атомарно увеличивает `current_spend` кампании и пропорционально уменьшает `reserved_amount` в таблице `campaign_quotas` с помощью нового SQL-запроса `DecreaseCampaignQuotaReserved`:
   ```sql
   UPDATE campaign_quotas
   SET reserved_amount = GREATEST(0, reserved_amount - $3),
       updated_at = NOW()
   WHERE shard_id = $1 AND campaign_id = $2;
   ```
   Использование `GREATEST(0, ...)` гарантирует соблюдение констреинта БД (`reserved_amount >= 0`) даже при микро-расхождениях округления.

2. **Защита от мертвых шардов (1.5.2):**
   В `ReconWorker` добавлен пинг каждого Redis-шарда. Если шард недоступен (таймаут/ошибка сети), воркер атомарно сбрасывает `reserved_amount = 0` в Postgres для всех кампаний этого шарда, если они не обновлялись более 60 секунд. Это освобождает заблокированный бюджет для других операций.

3. **Детектирование дрифта квот (1.5.3):**
   Введена математически точная формула проверки консистентности квот:
   $$\text{Drift} = \left| \text{reserved\_amount} - (\text{Redis\_Quota} + \text{Redis\_Sync} + \text{Redis\_Inflight}) \right|$$
   Если дрифт по кампании превышает размер чанка (`chunk_size`), `ReconWorker` логирует критическую ошибку (`QUOTA DRIFT DETECTED`) с детальным дампом всех состояний для алертинга в Grafana.

### Фаза 1.6: Локальный микро-кэш на трекере (Hot Campaign)
Для предотвращения деградации Redis при наплыве трафика на исчерпанные кампании разработан `LocalQuotaCache` (`internal/ads/quota_local.go`):
* **Структура данных:** Фиксированный массив из 4096 слотов (`localBlockSlot`). Каждый слот содержит ID кампании (`uuid.UUID`) и монотонное время блокировки (`blockedAt` в наносекундах).
* **Lock-Free Алгоритм:** Карта слотов не использует мьютексы. Слот выбирается по хэшу `crc32Castagnoli(campaign_id) % 4096`. Чтение и запись времени `blockedAt` выполняются атомарно через `sync/atomic`.
* **Zero-Allocation:** Метод `IsBlocked` принимает текущее монотонное время и выполняет только арифметические операции над примитивами. На горячем пути чтения **полностью отсутствуют выделения памяти в куче (0 B/op, 0 allocs/op)**.
* **Интеграция в фильтр:** Если Lua-скрипт возвращает код `3` (бюджет исчерпан), трекер мгновенно блокирует кампанию локально на 100 мс. Все последующие запросы в течение этого окна отклоняются на уровне Go без сетевых походов в Redis, снижая нагрузку на Redis до 0 RPS по этой кампании.

## 4. Анализ аллокаций, которые невозможно или нецелесообразно устранять

В ходе оптимизации горячего пути мы снизили аллокации `UnifiedFilter.Check` с 29 до **10 allocs/op**. Оставшиеся 10 аллокаций (выделяющие суммарно ~160 байт) распределены следующим образом:

1. **Внутренние буферы драйвера `go-redis` (6 аллокаций, ~96 байт):**
   * При отправке команды `EVALSHA` драйвер `go-redis` под капотом выполняет форматирование команды в протокол RESP. Это включает в себя аллокации слайсов аргументов внутри самого драйвера, запись в буферы сетевого соединения (`net.Conn.Write`) и парсинг ответа.
   * *Почему не фиксим:* Переписывание или глубокий форк драйвера `go-redis` нецелесообразен, так как текущие 6 аллокаций дают ничтожно малую нагрузку на GC и полностью перекрываются сетевым RTT до Redis.
2. **Парсинг UUID кампании из сетевого пакета (2 аллокации, ~32 байта):**
   * При получении запроса `/track` идентификатор кампании передается в виде строки или байт, которые преобразуются в `uuid.UUID`. Хотя сам `uuid.UUID` является фиксированным массивом `[16]byte` и не аллоцирует в куче, некоторые вспомогательные проверки библиотеки `github.com/google/uuid` могут приводить к временному escape-анализу.
   * *Почему не фиксим:* Это стандартное поведение парсинга сетевых идентификаторов, оптимизация которого через самописные структуры UUID усложнит кодовую базу без видимого прироста производительности.
3. **Метрики Prometheus (2 аллокации, ~32 байта):**
   * При вызове `metrics.TrackerLocalQuotaBlockTotal.WithLabelValues(cid).Inc()` библиотека Prometheus выполняет поиск счетчика в хэш-таблице по строковому представлению UUID кампании. Это приводит к аллокации строкового ключа для поиска.
   * *Почему не фиксим:* Локальная блокировка — это исключительный путь (fail path), срабатывающий только при исчерпании бюджета кампании. На счастливом пути (happy path) этот метод не вызывается, поэтому оптимизировать его аллокации не имеет смысла.

## 5. Заключение и План миграции (Runbook Cutover)

Система распределенных квот полностью готова к промышленной эксплуатации. Для безопасного развертывания рекомендуется следующий план:

1. **QUOTA_MODE=off (Legacy):**
   * Развертывание Management-сервиса с запущенным `QuotaManager`.
   * Трекеры работают в обычном режиме, списывая бюджет напрямую из `budget:campaign:{cid}`.
2. **QUOTA_MODE=shadow (Тестирование):**
   * Переключение трекеров в режим `shadow`.
   * Трекеры считывают `budget:quota` (если инициализирована) или `budget:campaign`, но не блокируют трафик при ошибках квот, логируя расхождения. Фоновый `QuotaManager` начинает наполнять квоты.
3. **QUOTA_MODE=live (Продакшн):**
   * Полный переход на распределенные квоты. Бюджеты списываются строго локально из квот на шардах кампаний.
