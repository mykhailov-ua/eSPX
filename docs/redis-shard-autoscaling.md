# Автоматический автоскейлинг и ребалансировка шардов Redis

Этот документ описывает архитектуру, конфигурацию и тестирование системы автоматического автоскейлинга и ребалансировки слотов в eSPX (REDIS.md Фаза 2.4).

---

## 1. Архитектура автоскейлинга

Автоскейлинг в eSPX работает на уровне **Control Plane (Postgres)** и использует механизм **Fixed Slot Map (FSM)**. 

В AdTech-системах с высокой нагрузкой (High RPS) классический подход с изменением формулы `slot % N` на лету приводит к огромному перераспределению ключей (~67% при переходе с 6 на 4 шарда) и деградации производительности. Вместо этого eSPX использует **динамическую миграцию слотов**:
1. Фиксированные 1024 слота распределены по шардам в Postgres.
2. При перегрузке конкретного шарда система автоматически переносит часть его слотов на наименее загруженные шарды.
3. Перенос происходит абсолютно бесшовно (без даунтайма) по схеме: `ACTIVE` → `MIGRATING` (копирование ключей через `DUMP`/`RESTORE REPLACE`) → `DRAINING` (активация новой карты слотов и очистка старых ключей).

### Сбор метрик (`ShardMetrics`)

Мониторинг ресурсов каждого шарда осуществляется через команду Redis `INFO` (разделы `memory`, `stats`, `cpu`):
* **MemoryPct:** Процент утилизации памяти относительно `maxmemory` (порог: `> 85%`).
* **OpsPerSec:** Текущий RPS на шарде (порог: `> 50 000 ops/sec`).
* **CPUUsage:** Утилизация CPU системными и пользовательскими процессами (порог: `> 80%`).
* **LuaP99Ms:** Время выполнения Lua-скриптов (порог: `> 15 ms`).

## 2. Алгоритм ребалансировки (`AutoscaleShards`)

Метод `AutoscaleShards` в `internal/management/service_shard_autoscaling.go` выполняет следующие шаги:

1. **Сбор метрик:** Опрашивает все настроенные шарды Redis через `ShardMetricsProvider`.
2. **Определение перегрузки:** Находит шард с наибольшей нагрузкой (`maxShard`), превышающий любой из установленных лимитов.
3. **Выбор цели:** Находит шард с наименьшей нагрузкой (`minShard`).
4. **Создание новой версии:** Создаёт черновик новой карты слотов (клонируя активную).
5. **Выбор слотов для миграции:** Выбирает подмножество слотов (по умолчанию 16), принадлежащих перегруженному шарду.
6. **Запуск миграции:**
   * Помечает слоты как `MIGRATING` в новой версии.
   * Регистрирует задачи миграции в Postgres.
   * Копирует данные кампаний для этих слотов на целевой шард.
   * Атомарно активирует новую версию карты слотов (все трекеры и Nginx мгновенно переключаются на новые пути).
   * Очищает старые ключи на исходном шарде (`DRAINING`).

## 3. Защита от дедлоков и гонок

При работе в распределенной среде и высокой конкурентности критически важно избегать дедлоков в базе данных.

### А. Сортировка блокировок (Sorted Locking)
В методе `AutoscaleBudgets` (перераспределение бюджетов между кампаниями одного клиента на основе CTR) блокировка кампаний в Postgres происходила в порядке "худшая кампания", затем "лучшая кампания". При конкурентных запросах это могло привести к классическому дедлоку (Thread 1 блокирует A → B, Thread 2 блокирует B → A).

**Решение:** Мы внедрили гарантированную сортировку блокировок по строковому представлению UUID кампаний:
```go
if worstID.String() < bestID.String() {
    worstLocked = q.GetCampaignForUpdate(worstID)
    bestLocked = q.GetCampaignForUpdate(bestID)
} else {
    bestLocked = q.GetCampaignForUpdate(bestID)
    worstLocked = q.GetCampaignForUpdate(worstID)
}
```
Это полностью исключает дедлоки при конкурентном изменении бюджетов.

### Б. Оптимистичные блокировки метаданных карты слотов
При создании новой версии карты слотов транзакция захватывает эксклюзивную блокировку метаданных `LockSlotMapMeta`, гарантируя, что только один поток может успешно создать и активировать новую версию, а остальные конкурентные потоки получат безопасную ошибку конфликта версий и завершатся без повреждения данных.

## 4. Покрытие хаос-тестами (Netflix-style Chaos)

Мы реализовали мощный набор хаос-тестов в `internal/management/shard_autoscaling_chaos_test.go`:

### 1. `TestChaos_ShardAutoscale_SuddenLoadSpike` (Резкий рост нагрузки)
* **Сценарий:** На Shard 0 имитируется внезапный всплеск нагрузки (CPU = 95%, Memory = 90%, Ops = 60k). Shard 1 свободен.
* **Проверка:** Автоскейлер успешно переносит слоты с Shard 0 на Shard 1, копирует данные бюджетов кампаний, активирует новую карту и удаляет старые ключи. Данные не теряются, SLA соблюдается.

### 2. `TestChaos_ShardAutoscale_ShuffledShards` (Перемешивание порядка шардов)
* **Сценарий:** Физический порядок шардов в массиве `s.rdbs` перемешивается (индекс 0 указывает на физический Shard 1, индекс 1 на Shard 0).
* **Проверка:** Автоскейлер корректно сопоставляет логические индексы слотов с физическими клиентами Redis, находит перегруженный физический инстанс и успешно завершает миграцию на свободный инстанс.

### 3. `TestChaos_ShardAutoscale_ConcurrentAutoscaleDeadlock` (Стресс-тест дедлоков)
* **Сценарий:** Запускается 4 параллельных воркера автоскейлинга, пытающихся одновременно ребалансировать слоты при высокой конкуренции за блокировки в Postgres.
* **Проверка:** Система не уходит в дедлок. Ровно один воркер успешно завершает миграцию и активирует новую версию, а остальные безопасно отклоняются с ожидаемыми конфликтами блокировок.

## 5. Как запустить и настроить

### Конфигурация в `.env` или `config.Config`

Добавьте следующие параметры для управления автоскейлингом:

```bash
# Включение автоматического мониторинга и ребаланса шардов
SHARD_AUTOSCALE_ENABLED=true

# Пороги срабатывания автоскейлинга
SHARD_AUTOSCALE_CPU_LIMIT=80.0
SHARD_AUTOSCALE_MEMORY_LIMIT_PCT=85.0
SHARD_AUTOSCALE_OPS_LIMIT=50000
SHARD_AUTOSCALE_LUA_P99_LIMIT_MS=15.0

# Количество слотов, мигрирующих за один шаг ребалансировки
SHARD_AUTOSCALE_SLOTS_TO_MIGRATE=16
```

### Запуск тестов

Для запуска хаос-тестов автоскейлинга выполните:

```bash
go test -v ./internal/management -run TestChaos_ShardAutoscale_
```
