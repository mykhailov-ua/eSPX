# Архитектура микросервисов eSPX: Критерии разделения и топология

В eSPX применяется прагматичный подход к разделению монолита на микросервисы. Домены изолируются не ради следования трендам, а на основе измеримых технических факторов: влияние на производительность горячего пути (hot path), требования информационной безопасности (PCI DSS, изоляция секретов) и минимизация зоны поражения (blast radius) при сбоях.

Этот документ описывает стратегию разделения, систему оценки целесообразности выделения сервисов и детальный разбор текущей топологии.

---

## 1. Базовый принцип: Изоляция Hot-Path и Cold-Path

Главная архитектурная граница в eSPX проходит между высокопроизводительным контуром (Hot Path) и контуром управления/бизнеса (Cold Path).

- **Hot-Path (tracker ×4, processor)**: Должен быть полностью избавлен от бизнес-логики. Здесь недопустимы операции блокирующего ввода-вывода (I/O) в реляционные БД, вызовы внешних API, сборка мусора сторонними SDK и тяжелая рефлексия. Задача этого контура — максимально быстрый прием трафика, фильтрация по локальным/Redis кэшам и сброс сырых событий в буфер.
- **Cold-Path (микросервисы бизнес-логики)**: Оперирует стандартными gRPC/HTTP протоколами, работает со сложными структурами данных и реляционной СУБД Postgres. Время ответа здесь измеряется десятками миллисекунд (а не микросекундами), что позволяет использовать стандартные идиоматические библиотеки.

## 2. Критерии разделения СУБД: Схемы против Баз Данных

Вопреки распространенному шаблону «один микросервис — один физический инстанс СУБД», eSPX использует **один экземпляр PostgreSQL и один экземпляр ClickHouse** для большинства сервисов.

### Логическое разделение на уровне схем
Изоляция данных обеспечивается на логическом уровне через схемы Postgres:
- Схема `public` — настройки рекламных кампаний, транзакционный outbox, дедупликация (`sync_idempotency`).
- Схема `auth` — пользователи, сессии, API-ключи.
- Схема `payment` — транзакции Stripe, логи вебхуков.
- Схема `billing` — инвойсы и налоговые профили.
- Схема `notifier` — очередь сообщений и статусы доставки.

### Почему это выгоднее на текущем этапе?
1. **Экономия ресурсов**: Отсутствует оверхед на запуск и поддержку 5–6 отдельных контейнеров PostgreSQL, что критично для локального окружения разработчиков и k3s-кластеров.
2. **Транзакционная целостность при интеграциях**: Некоторые смежные домены (например, зачисление платежа в балансовую книгу `balance_ledger`) могут использовать ссылочную целостность на уровне СУБД, избегая распределенных транзакций (2PC).
3. **Физическая изоляция как крайняя мера**: Для платежного сервиса (`payment`) выделен собственный контейнер СУБД (`db-payment`), так как к нему предъявляются жесткие требования аудита безопасности (PCI DSS). Для остальных сервисов изоляции схем в одном инстансе достаточно.

## 3. Матрица оценки целесообразности (Scoring System)

Для принятия решения о создании нового сервиса `cmd/<name>` или интеграции кода в качестве пакета внутри монолита (`cmd/management`) используется балльная система. Каждый критерий оценивается от 0 до 2 баллов:

| Критерий | Описание | 0 баллов | 1 балл | 2 балла |
| :--- | :--- | :--- | :--- | :--- |
| **H (Hot-path Isolation)** | Влияние на контур импорта | Нет связи с hot-path | Косвенная связь через общие ключи Redis | Прямой риск для производительности tracker |
| **E (External Ingress)** | Внешние сетевые вызовы | Только внутренние RPC | Внутренние вызовы + редкие исходящие запросы | Публичные эндпоинты, вебхуки, OAuth (Stripe API) |
| **S (Secret / Compliance)** | Безопасность и секреты | Нет секретов | Обычные DSN БД | Требования комплаенса (PCI DSS), токены Stripe/SMS |
| **F (Failure Blast Radius)** | Изоляция зоны сбоев | Падение не влияет на систему | Ограниченное влияние на панели управления | Риск каскадного сбоя, OOM или шторма повторов |
| **L (Load Profile)** | Профиль и характер нагрузки | Сопоставим с административным | Периодические пики | Постоянный опрос (polling) или тяжелая аналитика |
| **C (Caller Count)** | Количество клиентов API | 0-1 клиент | 2 клиента | Множество сервисов зависят от gRPC-контракта |
| **D (Data Ownership)** | Границы владения данными | Чтение/запись общих таблиц | Свои таблицы в общей схеме | Собственная схема БД и независимый цикл миграций |
| **O (Operational Indep.)** | Операционная независимость | Развертывается строго с хостом | Может быть отключен в dev-режиме | Опциональный запуск; система работает без него |
| **T (Team Lifecycle)** | Команда и цикл релизов | Единый релизный цикл | Разные ветки, общий релиз | Полностью независимый цикл развертывания |

### Расшифровка суммы баллов:
- **0–5 баллов**: Монолитная часть. Реализуется как пакет в `internal/<name>/`, запускается в рамках процесса `cmd/management` или `cmd/processor`.
- **6–10 баллов**: Модульный монолит. Пакет в `internal/` с возможностью запуска в отдельном фоновом потоке.
- **11+ баллов**: Выделенный микросервис `cmd/<name>`. Имеет собственный gRPC-порт, схему миграций СУБД и авторизацию вызовов по внутреннему токену `x-internal-token`.

## 4. Разбор текущей топологии сервисов eSPX

### 1. Management (`cmd/management`) — 10/18 баллов
*Административный хаб системы. Объединяет REST API панели управления, планировщики кампаний, pacing-контроллер, фоновую сверку и gRPC-сервер расчетов (Settlement gRPC).*
* **Обоснование**: Является центральной точкой управления. Объединение расчетного gRPC (:51053) с management позволило избежать лишнего gRPC-хопа и снизить сложность развертывания, так как расчеты напрямую пишут в общую схему балансов Postgres.

### 2. Auth (`cmd/auth`) — 14/18 баллов
*Сервис аутентификации пользователей и валидации PASETO/API токенов.*
* **Обоснование**: Высокие требования к безопасности секретов (хэши паролей, приватные ключи подписи токенов). Выделение в отдельный gRPC-сервис позволяет масштабировать его независимо, а также изолировать критическую схему данных.

### 3. Payment (`cmd/payment`) — 16/18 баллов
*Шлюз обработки платежей, интеграция со Stripe, прием внешних вебхуков.*
* **Обоснование**: Публичный порт для вебхуков Stripe (:8187) подвержен внешним штормам запросов. Изоляция токенов Stripe и соответствие требованиям PCI DSS делают этот сервис главным кандидатом на полную физическую автономию, включая собственный инстанс базы данных Postgres (`db-payment`).

### 4. Billing (`cmd/billing`) — 11/18 баллов
*Генерация периодических инвойсов на основе балансовой книги.*
* **Обоснование**: Процесс расчета налогов (`TaxCalculator`) и агрегации миллионов строк балансов за месяц создает тяжелую нагрузку на СУБД. Сервис изолирован на чтение и запись в своей схеме `billing` и вызывается по gRPC из панели управления.

### 5. Notifier (`cmd/notifier`) — 12/18 баллов
*Очередь уведомлений и интеграции с провайдерами (Telegram, Slack, SMTP, SMS).*
* **Обоснование**: Сетевые запросы к внешним API мессенджеров и почтовых шлюзов имеют непредсказуемую задержку. Вынесение логики отправки в отдельный воркер с собственными предохранителями (Circuit Breakers) гарантирует, что сбои в Telegram API не приведут к зависанию потоков бизнес-логики.

### 6. IVT Detector (`cmd/ivt-detector`) — 7/18 баллов (Батч-воркер)
*Сканирование логов ClickHouse на предмет фрода и аномалий.*
* **Обоснование**: По характеру нагрузки относится к тяжелым батч-процессам. Ему не требуется держать открытые RPC-порты. Он оформлен в виде отдельного периодически запускаемого бинарного файла, который вычисляет нарушителей и отправляет их IP в черный список через локальное HTTP API сервиса management.

### 7. Log Evacuator (`cmd/log-evacuator`) — 8/18 баллов (Системная утилита)
*Ротация логов трекера и эвакуация в локальный архив.*
* **Обоснование**: Утилита должна запускаться локально на узле трекера для прямого доступа к файловой системе (ввод-вывод). У нее нет сетевого API, она не обращается к базам данных и работает независимо по файлу чекпоинта.
