# Тестирование и CI/CD инфраструктура eSPX

Качество и стабильность eSPX обеспечиваются многоуровневой системой тестирования, интегрированной в CI/CD контур GitHub Actions. Тесты делятся на классические (модульные, интеграционные), перформанс-гейты (производительность и аллокации) и хаос-тесты (устойчивость к отказам инфраструктуры).

---

## 1. Пирамида и уровни тестирования

```
┌─────────────────────────────────────────────────────────┐
│ 4. ХАОС-ТЕСТЫ    Docker/testcontainers, SIGKILL, network│
├─────────────────────────────────────────────────────────┤
│ 3. ИНТЕГРАЦИЯ    Сквозной HTTP/gRPC цикл, СУБД/Redis    │
├─────────────────────────────────────────────────────────┤
│ 2. СМОУК-ТЕСТЫ   Проверка топологии, портов и версий    │
├─────────────────────────────────────────────────────────┤
│ 1. UNIT-ТЕСТЫ    Table-driven, parallel, zero-alloc     │
└─────────────────────────────────────────────────────────┘
```

### Уровень 1: Модульные тесты (Unit Tests)
- **Стандарт написания**: Все тесты бизнес-логики (проверки фильтров, парсинг OpenRTB) строятся по табличному принципу (Table-Driven Tests).
- **Параллелизм**: Тесты запускаются с вызовом `t.Parallel()` для минимизации общего времени сборки. В коде тестов исключается изменение глобального состояния.
- **Граничные значения**: Обязательный контроль поведения при `nil`-указателях, пустых значениях, `math.MaxInt64` и невалидном UTF-8.

### Уровень 2: Смоук-тесты инфраструктуры
Перед развертыванием приложения запускаются скрипты валидации окружения:
- `scripts/ci/check_deps.sh` — проверяет доступность внешних портов и версий ПО.
- `scripts/redis/verify_redis_topology.sh` — опрашивает Sentinel-ноды и подтверждает корректность репликации и роли мастер/реплика.

### Уровень 3: Интеграционные тесты
- **Отказ от моков**: Использование заглушек СУБД (например, `sqlmock`) запрещено. Все тесты, работающие с SQL или Redis, поднимают реальные базы данных в Docker через библиотеку `testcontainers-go`.
- **Конкурентные проверки**: Тесты финансового контура порождают не менее 20 параллельных goroutine с флагом гонки (`-race`) для проверки атомарности списаний.

### Уровень 4: Хаос-тестирование (Chaos Engineering)
Эксперименты базируются на каноническом цикле Netflix:
1. **Определение установившегося режима (steady state)**: Нормальный уровень RPS, задержки p99 < 80 мс, отсутствие ошибок 5xx.
2. **Формулирование гипотезы**: «Отключение 1 шарда Redis снижает пропускную способность только для 25% кампаний, остальные работают без изменения p99».
3. **Внедрение сбоя**: Воркер `testcontainers` убивает контейнер СУБД/Redis (SIGKILL) или ограничивает пропускную способность сети.
4. **Проверка инвариантов**: Равенство расходов `Redis + sync = Postgres`, отсутствие утечек памяти.
5. **Протокол `chaos_proof`**: Успешно прошедший хаос-сценарий выводит строку:
   ```text
   chaos_proof fault=<имя_сбоя> <ключ>=<значение>
   ```
   Скрипт `scripts/chaos/test_chaos.sh` считает количество уникальных подтверждений в логе. Если оно меньше `CHAOS_MIN_PROOFS = 28`, сборка в CI признается упавшей.

## 2. CI Pipelines в GitHub Actions

В репозитории настроены три основных пайплайна:

### 1. Основной контур (CI - `ci.yml`)
Запускается при пуше изменений. Выполняет следующие шаги:
- **Lint**: Проверяет стиль оформления кода и комментариев (`scripts/ci/check_comments.sh`), запускает `golangci-lint` через Makefile.
- **Full Test**: Запускает интеграционные тесты без флага `-short` (`scripts/ci/full_test.sh`), разворачивая полноценные контейнеры.
- **Chaos**: Выполняет хаос-тесты с контролем лимита `chaos_proof`.
- **Terraform Validate**: Инициализирует и валидирует декларативные манифесты развертывания локального (`envs/local`) и staging (`envs/staging`) окружений.

### 2. Перформанс-гейт (Performance Gate - `perf-gate.yml`)
Срабатывает при изменениях в горячем контуре (`internal/ads`, `internal/rtb`, LUA-скриптах, API).

Пайплайн работает в двух режимах в зависимости от типа раннера:
- **GitHub-hosted runner (Smoke-режим)**: Контролирует только **zero-allocation policy** (проверяет, что бенчмарки на горячем пути показывают ровно `0 allocs/op`).
- **Self-hosted runner (Strict-режим)**:
  1. Выполняет стабилизацию CPU с помощью скрипта `scripts/perf/stabilize_cpu.sh` (включает режим `performance governor` на процессоре для минимизации джиттера частоты ядра).
  2. Запускает бенчмарки на PR-ветке и ветке-предке (`main`).
  3. Использует утилиту `benchstat` для сравнения времени выполнения.
  4. При обнаружении статистически значимой регрессии по времени выполнения (> 5%) блокирует слияние PR.

## 3. Анализ кода компилятором (gcflags)

Для автоматического отсечения неэффективного кода на этапе сборки CI запускает компилятор со специальными флагами:

- **Escape-анализ**: Выявляет переменные, утекшие из стека в кучу (вызывающие лишние аллокации и работу GC):
  ```bash
  go build -gcflags="-m -m" ./internal/... 2>&1 | grep "escapes to heap"
  ```
- **Анализ инлайнинга**: Проверяет, встроились ли мелкие функции горячего пути непосредственно в место вызова:
  ```bash
  go build -gcflags="-m" ./internal/... 2>&1 | grep "can inline"
  ```
- **Проверка BCE (Bounds Check Elimination)**: Проверяет, что компилятор успешно удалил из машинного кода проверки выхода за границы слайсов в горячих циклах:
  ```bash
  go build -gcflags="-d=ssa/prove/debug=1" ./internal/...
  ```
  *Если в горячем цикле обнаруживается генерация инструкции проверки границ, сборка отклоняется, а код требует переписывания с добавлением BCE hints.*
