# Рефакторинг и интеграция стандарта OpenRTB 3.0 / AdCOM 1.0

## 1. Анализ стандарта OpenRTB 3.0 с точки зрения производительности в Go

Стандарт OpenRTB 3.0 разделяет транспортный уровень (Layer 3: Transaction Envelope) и доменную модель (Layer 4: Domain Model), в качестве которой выступает **AdCOM 1.0** (Advertising Common Object Model). 

Ниже представлен детальный анализ стандарта с точки зрения высокопроизводительной Go-разработки (целевой показатель трекера: **0 allocs/op** на горячем пути, p95 < 50 ms, p99 < 80 ms).

### Положительные стороны (The Good)
* **Стандартизация Protocol Buffers (Protobuf):** IAB Tech Lab стандартизировал protobuf-представление для OpenRTB 3.0. При использовании генераторов вроде `vtproto` с пулингом сообщений (`-features=pool`), разбор и сериализация выполняются с **0 allocs/op** и **0 B/op**.
* **Замена строк на Enums:** AdCOM заменяет множество текстовых полей из OpenRTB 2.x на целочисленные перечисления (например, `os`, `connectiontype`). Это позволяет парсить значения на уровне регистров процессора без выделения памяти в куче.

### Неприемлемые архитектурные решения для Hot Path (The Bad & Unacceptable)
* **Экстремальная вложенность JSON:** Структура запроса имеет глубокую вложенность: `openrtb` -> `request` -> `item` -> `spec` -> `placement` -> `display` -> `displayfmt`. Стандартный `encoding/json` на рефлексии генерирует сотни аллокаций на запрос, перегружая Garbage Collector (GC).
* **Полиморфизм и динамические интерфейсы:** Поля `spec` и `context` являются полиморфными (могут содержать разные доменные объекты). В Go это приводит к боксингу интерфейсов (`interface{}`) и выделению памяти в куче.
* **Текстовые идентификаторы:** Поля вроде `tagid`, `ua`, `ip` представлены строками, что требует аллокаций при стандартном парсинге JSON.

## 2. Реализация Zero-Allocation OpenRTB 3.0 парсера в eSPX

Для поддержки стандарта OpenRTB 3.0 без деградации производительности горячего пути был разработан специализированный парсер `ParseOpenRTB3Payload` в `internal/ads/openrtb_parse.go`.

### Ключевые особенности реализации:
1. **DFA / Byte-Slice Walk Scanner:** Вместо полного декодирования JSON-дерева через рефлексию, парсер выполняет линейный проход по байтовому слайсу за один проход ($O(N)$), извлекая только необходимые для аукциона поля:
   * `"flr"` (floor price) внутри первого элемента массива `item` $\rightarrow$ `MinBid`.
   * `"type"` внутри объекта `device` $\rightarrow$ `DeviceType`.
   * `"category_mask"` $\rightarrow$ `CategoryMask`.
2. **Zero-Alloc Decimal Parser (`parseDecimalMicro`):** Парсит дробные числа с плавающей точкой (например, `"flr": 1.50`) напрямую в целочисленные микро-единицы (`int64` умноженный на 1,000,000) без аллокаций строк или использования `strconv.ParseFloat`.
3. **Обратная совместимость:** Метод `buildRtbTargeting` в `internal/ads/rtb_track.go` сначала пытается разобрать payload как OpenRTB 3.0. Если ключ `"openrtb"` не найден, он прозрачно переключается на легаси-парсер плоского JSON.

## 4. Сводный отчет по выполненным работам в сессии (Фазы 1.1 – 1.4)

В рамках текущей сессии была полностью реализована архитектура распределенных квот (**Distributed Quotas**) и проведена глубокая оптимизация горячего пути фильтрации.

### Фаза 1.1: Схема Postgres и Финансовый Инвариант
* **Миграция:** Создана таблица `campaign_quotas` (`migrations/00027_campaign_quotas.sql`) для отслеживания выделенных квот.
* **Финансовый Инвариант:** В транзакции `ReserveChunk` (`quota_repo.go`) реализована строгая проверка:
  $$\text{Spent\_In\_DB} + \sum \text{Reserved\_In\_Redis} \le \text{Budget\_Limit}$$
  Это полностью исключает оверспенд при асинхронной синхронизации.
* **Идемпотентность:** Дедупликация запросов на резервирование через таблицу `sync_idempotency` с префиксом `quota:`.

### Фаза 1.2: Lock-Free Шардирование
* **Lock-Free чтение:** Карта слотов (`slotTable`) перенесена в `atomic.Value` в `sharding.go`. Доступ на чтение выполняется без мьютексов, каналов и системных вызовов.
* **CRC-32 Castagnoli:** Реализовано аппаратное ускорение вычисления хэша слота (SSE4.2/ARMv8) с generic-табличным фолбеком.

### Фаза 1.3: Redis Keys и Логика в `unified-filter.lua`
* **Refill Lock:** В скрипт `unified-filter.lua` добавлены ключи `budget:quota:{cid}`, `budget:refill_lock:{cid}` и `budget:refill_needed`.
* **Защита от Thundering Herd:** При падении квоты ниже порога (20% от размера чанка) скрипт атомарно устанавливает замок рефилла через `SET NX EX 10`. Только первый запрос добавляет ID кампании в `refill_needed`. Остальные запросы на горячем пути не спамят базу данных сигналами рефилла.
* **Dual-Read:** Поддерживается плавный переход: если ключ квоты отсутствует, скрипт считывает исторический ключ `budget:campaign:{id}`.

### Фаза 1.4: Оптимизация аллокаций горячего пути
* **Pooled EVALSHA (`redis_eval_pooled.go`):** Созданы пулы для wire-`[]any` аргументов и структур `redis.Cmd`. С помощью `unsafe.Pointer` выполняется in-place сброс полей `redis.Cmd` без перевыделения памяти, обходя аллокации `go-redis`.
* **Zero-Alloc Deadlines:** Дедлайны перенесены непосредственно в структуру события `domain.Event` (`FilterDeadlineMono`), полностью устранив аллокации `context.WithValue`.
* **Pre-bound Prometheus Metrics:** Введен бакетинг кампаний (`sampledCampaignBuckets = 256`). Все счетчики для бакетов создаются заранее, устраняя динамическое форматирование UUID в строки и поиск в хэш-таблицах Prometheus на горячем пути.
* **Zero-Alloc Time Boxing:** Создана переменная `cachedUnixMilliAny` типа `atomic.Value`, обновляемая фоновой горутиной раз в миллисекунду. Это устранило боксинг времени при передаче аргументов в Lua.
