## 1. Обзор архитектуры

В текущей реализации RTB не является отдельным OpenRTB-сервером. Он вызывается напрямую в `processTrack()` перед `FilterEngine.Check`:

```
Ingest (/track, gnet)
  → ensureIngestGeo (MaxMind → evt.GeoHash)
  → applyRtbAuction (в зависимости от режима)
       off    → пропуск
       shadow → RunAuctionEval + метрики (кампания клиента не меняется)
       live   → RunAuction + списание бюджета → замена evt.CampaignID
  → FilterEngine (гео, фрод, единый Lua-фильтр, …)
  → Поток расчёта и сохранения (Settlement)
```

### Структура пакета

| Компонент | Путь | Назначение |
|-----------|------|------------|
| Ядро аукциона | `internal/rtb/` | Реестр кампаний, ранжирование, биллинг, `BudgetStore`, снимки состояния |
| Связка с трекером | `internal/ingestion/rtb_*.go` | Синхронизация каталога, интеграция с `/track`, разбор OpenRTB, shadow-метрики |
| Панель управления | `internal/management/handler_rtb.go`, `service_rtb_deals.go`, `service_bid_floor.go` | CRUD для сделок (PMP Deals), валидация bid-запросов, оптимизатор минимальных цен (Floors) |
| Конфигурация | `internal/config/rtb.go`, `env.go` | Режимы `RTB_MODE`, выбор источника бюджета, таргетинг-индекс |

### Потоки данных (Cold Path vs Hot Path)

| Путь | Точки входа | Операции |
|------|-------------|----------|
| **Холодный** | `SyncRtbCatalog`, `ReloadRtbDeals`, `UpdateCampaigns` | Пересчёт гео-шардов, сборка SoA-массивов, инвертированный индекс, предварительная сортировка по Score |
| **Горячий** | `RunAuction` → `rankCandidates` | Сканирование готовых SoA-слайсов, списание бюджета через CAS |

Потоки чтения каталога никогда не блокируют запись: `Registry.catalog` — это `atomic.Pointer[catalogSnapshot]`, указатель подменяется при пересборке.

## 4. Низкоуровневая инженерия горячего пути (Hot-Path)

Основано на правилах [GO.md](./GO.md) (§§4–7). RTB служит эталонным примером использования структуры массивов (Structure of Arrays, SoA) и экстремальной оптимизации ассемблерного кода.

### 4.1 Структура массивов (SoA) и локальность кэша CPU

**Проблема подхода Array of Structures (AoS):** срез `[]Campaign{bid, ctr, mask, …}` заставляет процессоры загружать в кэш ~80+ байт на каждого кандидата, хотя на первом этапе нужны только `DeviceMask` и `Bid`.

**Решение:** двухуровневая структура SoA:

1. **Реестр шардов** — `CampaignAuctionRegistry` хранит параллельные срезы (`Bids []int64`, `DeviceMasks []uint8`, …).
2. **Базовый SoA-контейнер (Bucket)** — `candidateBucketSoA` копирует горячие поля **в порядке обхода** для конкретного гео/таргетинг бакета.

Фоновый процесс (`appendBucketCandidate`) формирует строки бакета заранее. Цикл аукциона никогда не обращается к `reg.Bids[i]` через индекс — чтение идёт строго из непрерывных массивов `bids[pos]`, `deviceMasks[pos]` и т.д.

**Эффект для кэша L1/L2:** маски типов устройств и лимитов (`uint8`) упаковываются по ~64 значения в одну 64-байтовую строку кэша CPU. Последовательное чтение `pos++` идеально обрабатывается аппаратным префетчером (Hardware Prefetcher). Отсутствие косвенной адресации вида `catalogIdx[pos]` → `reg.*[i]` устраняет простои ядра из-за цепочек зависимых загрузок (Pointer Chasing).

### 4.2 Исключение проверок выхода за границы (Bounds-Check Elimination, BCE)

Go вставляет инструкцию `runtime.panicIndex` при каждом обращении к `slice[i]`, если компилятор не может гарантировать, что `i < len(slice)`.

**Шаблон проверки в `rankCandidates`:**

```go
// 1. Проверяем весь диапазон бакета один раз при входе
if bucketStart < 0 || bucketEnd < bucketStart || bucketEnd > soa.len() {
    return ..., NoBidCorruptCatalog
}
// 2. Создаём подслайсы фиксированной длины — проверки выносятся за пределы будущего цикла
catalogIdx := soa.CatalogIdx[bucketStart:bucketEnd]
bids := soa.Bids[bucketStart:bucketEnd]
// ...
// 3. Цикл использует len(catalogIdx) и pos — вызовов panicIndex при чтении нет
for pos := 0; pos < len(catalogIdx); pos++ {
    bid := bids[pos]
}
```

**Проверка:** выполнение `go test -gcflags=-d=ssa/check_bce/debug=1` или `go tool objdump -S` на функции `rankCandidates` подтверждает отсутствие `CALL runtime.panicIndex` внутри тела цикла (зафиксировано в M18 §7).

**Правило для новых циклов RTB:** выполнять проверку длины буфера при входе (`if len(buf) <= i { return ErrMalformed }`), избегая повторных проверок внутри тела цикла.

### 4.3 Предотвращение ложного обмена кэшем (False Sharing) в `AlignedBudget`

Параллельные аукционы для разных кампаний списывают деньги из разных слотов бюджета. Обычный массив `[]int64` укладывает 8 слотов в одну 64-байтовую строку кэша. Операция CAS над слотом A на одном ядре процессора делает недействительной всю строку кэша для слота B на другом ядре (протокол согласования кэшей MESI).

```go
type AlignedBudget struct {
    Value int64
    _     [7]int64  // выравнивание до 64 байт (отдельная строка кэша на слот)
}
```

Каждая атомарная инструкция `CompareAndSwapInt64` работает со своей изолированной строкой кэша. Плата за скорость: увеличение расхода памяти под бюджеты в 8 раз, что абсолютно незначительно при числе кампаний < 100k.

**Настройки каталога:** поля `clearingMode`, `targetingIndexEnabled` используют `atomic.Uint32` / `atomic.Bool` — значение считывается 1 раз на весь аукцион, а не для каждого кандидата.

### 4.4 Предсказание ветвлений (Branch Prediction) и безветвистые (Branchless) вычисления

Ошибки предсказания ветвлений (Branch Misprediction Penalty) на современных процессорах с глубоким конвейером (14–20 стадий) стоят до 15–20 тактов сброса конвейера. В цикле аукциона с проверкой 500 кандидатов это слагаемое может составлять до 60% общего времени выполнения.

#### Анализ предсказываемости ветвей CPU (BTB / TAGE)

| Ветвь | Предсказуемость | Примечание |
|-------|-----------------|------------|
| Фильтр устройств/категорий `(deviceMasks[pos] & deviceType) == 0` | Высокая | Побитовое И, без map; тип устройства неизменен в рамках запроса |
| Проверка лимитов (Pacing) `pacingOpen[pos] == PacingClosed` | Высокая | Стабильно для кампании |
| Ранний выход по score | Высокая | Неверный прогноз — ровно 1 раз за аукцион |
| Исчерпание бюджета `LoadBudget < bid` | Средняя | Падает при активном расходе бюджета |
| Повреждение каталога | Крайне редкая | Аварийные выходы; компилятор выносит в хвост функции |

#### Безветвистый отбор (Branchless CMOV) и побитовая фильтрация

Для устранения условных переходов `JMP`/`JNE` внутри цикла сканирования применяются две техники:

1. **Побитовые маски вместо `if/else`:**

   ```go
   // Вместо: if (deviceMask & reqDevice) != 0 && (categoryMask & reqCategory) != 0
   // Используем единое побитовое вычисление без ветвления:
   match := uint64((deviceMasks[pos] & reqDevice) | (categoryMasks[pos] & reqCategory))
   ```

2. **Безадресные инструкции условий (`CMOV` в x86 / `CSEL` в ARM64):**

   Компилятор Go (SSA backend) генерирует `CMOVQGT` / `CMOVQLE` при обновлении `maxScore` и `winnerIdx`, если условие не содержит вызовов функций:

   ```go
   // Генерирует CMOVQGT вместо JLE/JMP
   if score > maxScore {
       maxScore = score
       winnerIdx = pos
   }
   ```

**Инвариант монотонности:** сортировка `sortBucketSoA` упорядочивает данные по `effectiveScore` (desc), затем по `Weight`, затем по `Bid`. Ранний выход через `break` корректен только при полной отсортированности бакета — пропускать сортировку при фоновой пересборке каталога запрещено.

### 4.5 Глубокий анализ ассемблера Go (Plan 9 ASM) и сокращение инструкций

Ассемблер Go использует синтаксис Plan 9 (псевдо-регистры `FP`, `SP`, `SB`, `PC`, мнемоники `MOVQ`, `LEAQ`, `ADDQ`).

#### 1. Закрепление переменных в регистрах (Register Pinning)

| Симптом в objdump | Решение |
|-------------------|---------|
| Повторяющиеся `MOVQ ... (SP)` внутри цикла | Объединение полей в битовые маски (`uint64`), локальные переменные-кэши, вынос неизменяемых параметров за пределы цикла |

#### 2. Инлайнинг функций и устранение стековых кадров

Вызов неинлайненной функции в цикле: сбрасывает регистры на стек, вставляет `CALL runtime.morestack`, блокирует BCE и векторные инструкции. В RTB hot-path все функции расчётов (`effectiveScore`, `applyReserve`) имеют **inline-cost < 80** и инлайнятся без стекового кадра (`NOSPLIT`).

#### 3. Шаблон Plan 9 ASM для `rankCandidates`

```assembly
// Начало цикла rankCandidates
TEXT ·rankCandidates(SB), NOSPLIT, $0-48
    ...
Loop:
    CMPQ    CX, SI              // pos < len(catalogIdx) ?
    JGE     Done                // Выход из цикла
    MOVQ    (AX)(CX*8), R8      // bids[pos] — прямое чтение из SoA
    MOVBLZX (BX)(CX*1), R9      // deviceMasks[pos]
    ANDQ    R10, R9             // deviceMask & reqDevice
    TESTQ   R9, R9
    JZ      Next
    IMULQ   R11, R8             // bid * ctr
    CMPQ    R8, R12             // score > maxScore ?
    CMOVQGT R8, R12             // maxScore = score (Branchless!)
    CMOVQGT CX, R13             // winnerIdx = pos (Branchless!)
Next:
    INCQ    CX
    JMP     Loop
Done:
```

### 4.6 Zero-Copy и арены памяти (Arena Locality)

На горячем пути RTB **абсолютно запрещены** динамические выделения памяти в куче (`runtime.newobject`, `runtime.makeslice`).

#### 1. Zero-Copy парсинг кадра gnet

При поступлении пакета OpenRTB/JSON обработчик `gnet` выделяет кольцевой буфер на уровне ОС.

- Для полей (`flr`, `deal_id`, `category`) — поиск смещений через `bytes.IndexByte` и сканер `parseDecimalMicro`.
- Преобразование `[]byte` → `string` без копирования:

  ```go
  func bytesToStringUnsafe(b []byte) string {
      return *(*string)(unsafe.Pointer(&b))
  }
  ```

- **Гарантия жизненного цикла:** строка существует **только** во время `processTrack()`; не сохраняется в долгоживущие структуры без `copy()`.

#### 2. Работа в пределах одной арены памяти / sync.Pool

- Пулы `sync.Pool` с фиксированной ёмкостью (например `[512]uint32`).
- Очистка буферов сбросом длины (`slice = slice[:0]`) без освобождения базового массива.
- Временные результаты (`AuctionResult`, `BidRequest`) передаются по значению — в регистрах CPU или на стеке.

### 4.7 Атомарные снимки состояния (исключение проблемы Thundering Herd)

| Структура | Механизм |
|-----------|----------|
| `Registry.catalog` | `atomic.Pointer[catalogSnapshot]` — читатели берут указатель 1 раз |
| `DealIndex.snap` | Аналогично; безопасный параллельный поиск при фоновой перезагрузке |
| `DealFloorCache.snap` | `atomic.Pointer[map[string]int64]` |
| `BudgetStore.budgets` | `atomic.Pointer[budgetSlice]`; расширение массива только на холодном пути |

**Главное правило горячего цикла:** локальные переменные `store := registry.store` и `regCount := reg.Count` захватываются 1 раз перед циклом; вызывать `atomic.Load` внутри цикла перебора кандидатов запрещено.

**Атомарный CAS бюджета:** `checkAndSpendOn` выполняет CAS-цикл по адресу `&slice.data[idx].Value` — конкуренция за замок идёт на уровне отдельной кампании, а не всего сервиса.

### 4.8 Разбор OpenRTB 3.0 с нулевым выделением памяти

`ParseOpenRTB3Payload` использует `bytes.Index` прямо по буферу кадра gnet (`evt.Payload`):

- Без `json.Unmarshal` и без создания `string` для числовых полей.
- Ручной обход цифр в `parseDecimalMicro` для `flr`.
- `ParseDealID` делает аллокацию `string(...)` **только 1 раз**, если в запросе присутствует `deal_id`.

`buildRtbTargeting` выполняется перед аукционом и сохраняет 0 аллокаций для обычных запросов (бенчмарк: `BenchmarkBuildRtbTargeting_OpenRTB3`).

### 4.9 Монотонные часы и сэмплирование метрик

```go
//go:linkname monotonicNano runtime.nanotime
func monotonicNano() int64
```

| Оптимизация | Описание |
|-------------|----------|
| Монотонный таймер | Не подвержен прыжкам из-за синхронизации NTP |
| `metricsEnabled` | Атомарная проверка — выключен во время бенчмарков |
| `rtbAuctionMetricsSampleMask = 127` | Метрики пишутся только для 1 из 128 аукционов |
| `bindAuctionMetrics()` | Указатели Prometheus привязаны заранее — без `WithLabelValues` на горячем пути |

### 4.10 Анализ побега на кучу (Escape Analysis) и инлайнинг

Целевые показатели (по результатам анализа в M18):

| Тип / функция | Поведение |
|---------------|-----------|
| `BidRequest`, `AuctionResult` | Выделяются исключительно на стеке |
| `RunAuction` | Полностью инлайнится в `applyRtbAuction` |
| `rankCandidates` | `req` и `soa` не убегают на кучу; `registry` — ссылки на метаданные без heap-объектов |

**Запрещено:** передача конкретных структур через `interface{}` в коде аукциона; использование замыканий внутри циклов `for`.

### 4.11 Регламент анализа ассемблерного кода (Plan 9 / Assembly)

При внесении изменений в `rankCandidates` или `CheckAndSpendAll`:

```bash
go test -run='^$' -bench=BenchmarkAuction -benchmem ./internal/rtb/
go test -gcflags="-m -m" ./internal/rtb/ 2>&1 | rg 'rankCandidates|RunAuction'
go test -c -o /tmp/rtb.test ./internal/rtb/
go tool objdump -S -l /tmp/rtb.test | rg -A30 'rankCandidates'
```

| Симптом в ассемблере | Действие |
|----------------------|----------|
| `CALL runtime.panicIndex` в цикле | Сбой BCE; требуется явная проверка границ перед циклом |
| `CALL runtime.morestack` в цикле | Увеличение стека / вызов неинлайненной функции |
| `LOCK` по глобальному адресу | Ложный обмен кэшем или атомарная операция в цикле; вынести чтение за цикл |
| Сброс `maxScore`/`winnerIdx` на каждой итерации | Переполнение регистров CPU; упростить тело цикла |

### 4.12 Лимиты времени выполнения (SLA)

| Компонент | Цель (p99) |
|-----------|------------|
| Обработка запроса `/track` | < 80 мс |
| `RunAuction` | < 15 мкс |
| Проверено кандидатов | < 500 |
| Единый Lua-фильтр в Redis (на шард) | < 10 мс |

При текущей скорости аукциона ~25–112 нс/оп (по бенчмаркам) узким местом является не CPU, а **количество кандидатов** и **последующие вызовы Redis**. Индекс таргетинга `RTB_TARGETING_INDEX` сокращает ширину поиска, а предсортировка избавляет от лишних итераций после нахождения победителя.

## 5. Развертывание и управление бюджетами

### 5.1 Матрица режимов работы

| `RTB_MODE` | Аукцион | Списание бюджета | `campaign_id` |
|------------|---------|------------------|---------------|
| `off` | Не выполняется | — | Не меняется |
| `shadow` | `RunAuctionEval` | Нет | Не меняется |
| `live` | `RunAuction` | Согласно authority | Заменяется UUID победителя |

### 5.2 Источники контроля бюджета (Budget Authority)

| Authority | `skip_budget` в Lua | Источник правды | Списание |
|-----------|---------------------|-----------------|----------|
| `redis` (по умолчанию) | `false` | Redis | `unified-filter.lua` |
| `rtb` | `true` | `BudgetStore.CheckAndSpendAll` | Redis — зеркало + периодическая сверка |

**Порядок переключения в продакшене:**

1. `shadow` до стабилизации совпадения победителей
2. `live` + `redis`
3. `live` + `rtb` при отсутствии алертов расхождения бюджетов

## 6. План развития (Roadmap)

Этапы расставлены с учётом требований рынка (OpenRTB 2.6, запуск PMP-сделок, проверка цепочек поставщиков) и устранения отставаний eSPX.

| Этап | Приоритет | Задачи | Фокус |
|------|-----------|--------|-------|
| 1 | P0 | R1–R5 | Замыкание контура измерений и PMP-сделок |
| 2 | P0 | R6–R9 | Поддержка протокола OpenRTB 2.6 |
| 3 | P1 | R10–R16 | Оптимизация доходности и качества ранжирования |
| 4 | P1 | R17–R20 | Безопасность, цепочки поставок и защита от фрода |
| 5 | P2 | R21–R26 | Расширение возможностей таргетинга |
| 6 | P2–P3 | R27–R31 | Платформа и эксплуатация |

### Этап 1 — Замыкание контура измерений и PMP-сделок (P0)

| ID | Задача |
|----|--------|
| **R1** | Запись исходов сделок (`rtb_deal_outcomes`) в ClickHouse. Фоновый путь: кольцевой буфер с потерей → пакетная ставка в CH (по аналогии с `FraudStreamWriter`). Поля: `deal_id`, `outcome`, `floor_micro`, `created_at`. Точка интеграции: `recordRtbShadowAuction` + `applyRtbAuction`. 0 аллокаций на горячем пути |
| **R2** | Проверка метаданных PMP-сделок. Расширить `BidRequest` полями `DealID` + `SeatCount` (на стеке). В `rankCandidates`: поиск по `DealIndex`; отказ при `geo_mask & reqGeo == 0`, `cat_mask & category == 0`, `PacingClosed`. Проверки **побитовые**, по возможности без ветвлений |
| **R3** | Заполнение `ReserveMicro`. `BuildRtbInputsFromRegistry` ← чтение колонки из кампании. Горячий путь не меняется; логика в `applyReserve` |
| **R4** | Включение `RTB_TARGETING_INDEX` по умолчанию. Нагрузочное тестирование (сканирование < 500 кандидатов на p99); включить после хаос-тестов |
| **R5** | Автоматический чек-лист готовности к Live-режиму. Эндпоинт: `RtbShadowDiffForWindow` + `ad_rtb_budget_reconcile_high` → `ready: bool` + причины |

### Этап 2 — Поддержка протокола OpenRTB 2.6 (P0)

| ID | Задача |
|----|--------|
| **R6** | Быстрый парсер OpenRTB 2.6 (`openrtb26_parse.go`): `imp[0].bidfloor`, `imp[0].pmp.deals[]`, `device.devicetype`, `site`/`app`, `source.ext.schain`. Только байты, BCE, без `encoding/json` на горячем пути |
| **R7** | Отдельный gnet-маршрут `/openrtb/bid`. Парсинг → `buildRtbTargeting` → `RunAuction` → минимальный `BidResponse`. Проброс `tmax` в монотонный дедлайн |
| **R8** | Генерация Bid-ответа. Фиксированный буфер `[]byte`, без `fmt.Sprintf`. Цена из микро-единиц через статический буфер цифр |
| **R9** | Контроль таймаута `tmax`. `BidRequest.DeadlineMono int64`; проверка `monotonicNano() > deadline` каждые N кандидатов (например, 32) |

### Этап 3 — Оптимизация доходности и качества ранжирования (P1)

| ID | Задача |
|----|--------|
| **R10** | Мульти-страновой таргетинг. Фоновый путь: отдельные строки каталога под каждую страну |
| **R11** | Учёт гибридных весов. `HybridBalancer` → `Weight` в `RtbCampaignInput` (сейчас `1`) |

| **R13** | Быстрые фильтры до аукциона: гео-маска, Circuit Breakers перед `RunAuction` |

## 7. Чек-лист для изменений на горячем пути (Hot-Path PR)

Любой PR с изменениями горячего пути RTB должен проходить проверки:

```bash
go test ./internal/rtb/... -short
go test -run='^$' -bench=BenchmarkAuction -benchmem ./internal/rtb/
make test-alloc-gate    # при изменении кода обработки /track в ingestion/rtb
go test -gcflags="-m" ./internal/rtb/ 2>&1 | rg 'does not escape'
```

| Критерий | Требование |
|----------|------------|
| Выделение памяти | 0 B/op, 0 allocs/op в `BenchmarkAuction` |
| Проверки границ (BCE) | Нет `panicIndex` в цикле `rankCandidates` (objdump) |
| Атомики | Нет `atomic.Load` внутри цикла перебора кандидатов |
| Интерфейсы | Нет `interface{}` и замыканий на пути аукциона |
| Строки | Нет конкатенаций через `+` на каждый запрос |
| Метрики | Предварительно привязанные счётчики; сэмплирование гистограмм |
| Хаос-тесты | `go test -run TestChaos ./internal/rtb/...` при правках каталога или бюджетов |
